# ניהול תקלות (Bug Tracking & Resolution)

**מסמך:** `BUG_REPORT.md`  
**יוצר:** סוכן פיתוח (תיעוד תקלה אוטומטי בזמן אמת)  
**מאשר ומתקן:** אדם (מפתח / QA / Tech Lead)  
**סוג אפיון:** ניהול תקלות  

---

## הערת פתיח – שימוש אוניברסלי

מסמך זה הוא חלק ממערך תבניות גנרי של **™Docs-as-System**,  
ומשמש בסיס לתיעוד, בקרה ופיתוח בכל תחום תוכנה –  
Web, Mobile, Backend, Cloud, Data, AI, Embedded או Multi-Agent.  

הוא מניח מודל עבודה שבו הסוכן (AI) הוא המבצע בפועל  
והאדם הוא בעל הסמכות, האישור והבקרה.  
הסוכן פועל תחת מדיניות מתועדת, מתעד כל פעולה,  
ומבצע Commits מקומיים בלבד במערכת ניהול גרסאות (Git או דומה).  
האדם מאשר את השינויים ומבצע את ה-Merge הסופי.  

**עקרון היסוד:**  
הסוכן מיישם → האדם מאשר → Git שומר את האמת.  

---

## מטרת המסמך

לתעד כל תקלה שהתגלתה במערכת במהלך פיתוח, בדיקה או הפעלה,  
ולהבטיח שהתגובה האנושית והאוטומטית מתבצעת בצורה עקבית ומתועדת.  

המסמך מהווה **מקור אמת אחיד** בין הסוכן (שמזהה ומדווח)  
לבין האדם (שמאבחן, מתקן ומאשר את הפתרון).

## מבנה תקלה (Bug Entry)

| שדה | ערך למילוי | הסבר |
|------|--------------|--------|
| מזהה תקלה | BUG-XXXX | מזהה ייחודי אוטומטי |
| תאריך גילוי | (למילוי) | תאריך + שעה |
| שלב שבו נמצאה | פיתוח / בדיקה / פריסה / ייצור | שלב גילוי התקלה |
| מקור התקלה | (למילוי) | רכיב או מודול רלוונטי |
| תיאור התקלה | (למילוי) | מה קרה בפועל |
| תוצאה צפויה | (למילוי) | מה היה אמור לקרות |
| חומרה | נמוכה / בינונית / גבוהה / קריטית | לפי השפעה |
| הסתברות | נמוכה / בינונית / גבוהה | לפי תדירות |
| סטטוס נוכחי | פתוחה / בטיפול / תוקנה / נסגרה | מצב עדכני |
| אחראי אנושי | (שם) | מי אחראי לטיפול |
| מדווח (סוכן) | (שם או מזהה סוכן) | מקור הדיווח |

## שלבי טיפול בתקלה (Bug Lifecycle)

| שלב | תיאור פעולה | אחראי | הערות |
|-------|----------------|----------|----------|
| זיהוי | הסוכן מזהה חריגה, לוג או כשל הרצה | סוכן | דיווח אוטומטי |
| אימות | האדם בודק אם מדובר בתקלה אמיתית | אדם | אישור נדרש |
| אבחון | הסוכן מנתח הקשר, קבצים ורשומות | סוכן | סיוע באיתור סיבה |
| תיקון | האדם מבצע שינוי בקוד או בקונפיגורציה | אדם | מתועד אוטומטית |
| בדיקת תיקון | הסוכן מריץ בדיקות Self-Check | סוכן | נדרש אישור אנושי |
| סגירה | QA / Tech Lead מאשרים את התיקון | אדם | נחתם רשמית |

## תיעוד נלווה

| סוג קובץ | תיאור | מיקום |
|------------|----------|----------|
| Screenshot / Log | תצלום מסך או קובץ לוג רלוונטי | (נתיב / קובץ) |
| Commit Reference | קישור ל-Commit או PR מתקן | (למילוי) |
| Test Report | דוח בדיקות המאשר את התיקון | (למילוי) |
| Decision Reference | קישור להחלטה עסקית / טכנית רלוונטית | (למילוי) |

## סיווג תקלות (Bug Classification)

| קטגוריה | תיאור | דוגמה |
|-----------|---------|---------|
| לוגית | שגיאה בחוק עסקי או תהליך לוגי | חישוב מחיר לא נכון |
| תשתיתית | בעיה בתקשורת, תור, DB, API | חיבור נקטע או Timeout |
| ממשק משתמש | שגיאת תצוגה / UX | כפתור לא מגיב |
| אבטחה | בעיית הרשאה או זליגת מידע | Token לא תקף |
| תיעוד | אי-עקביות בין קוד למסמך | תיעוד חסר או שגוי |

## מדדי איכות ניהול תקלות

| מדד | יעד | מקור נתונים | הערה |

## Agent Collaboration Note

- הסוכן מזהה תקלות בזמן אמת ומפיק דיווחים מיידיים.  
- מבצע ניתוח ראשוני, כולל השוואת גרסאות, קבצים ותיעוד.  
- מתייג אוטומטית את חומרת התקלה ומציע פעולת תיקון.  
- האדם מאשר, מבצע תיקון ומוודא סגירה מלאה.  

האינטראקציה נשמרת בתיעוד רציף המאפשר ניתוח ולמידה עתידית.
